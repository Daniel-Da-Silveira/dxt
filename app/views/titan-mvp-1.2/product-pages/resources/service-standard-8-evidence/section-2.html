2. Continued research and iteration based on feedback, data, and metrics

The Defra Forms Designer demonstrates continuous iteration and improvement through systematic research, feedback collection, and data-driven decision making.

Example 1: Systematic User Research Programme

Situation:
The service needed to continuously improve based on real user feedback and usage data, not assumptions.

Task:
Establish a systematic programme of user research that informs iterative improvements throughout the service lifecycle.

Action:

  • Conducted 17 rounds of user research between May 2024 and October 2025
  • 77 research iterations across 32 features, with an average of 2.4 iterations per feature
  • 27 features (84%) tested and validated with users
  • Over 70 participants including content designers, proxy users, internal stakeholders, and Private Beta users
  • Each research round builds on previous findings, with issues identified in one round addressed and validated in subsequent rounds
  • Research findings systematically documented in design history

Result:

  • Continuous improvement cycle established and maintained
  • Every major change informed by user research and feedback
  • Service remains relevant and meets user needs as they evolve
  • Design history provides comprehensive record of iterations and improvements

Example 2: Using Performance Metrics to Inform Improvements

Situation:
User feedback about the "save and exit" feature indicated some users were being locked out, but it was unclear if this was a widespread issue or an isolated incident.

Task:
Use performance metrics alongside user feedback to understand the real scope of issues and prioritise improvements.

Action:

  • Analysed logs to understand actual usage patterns
  • Found 94 people successfully resumed a saved form on the same day as the complaint
  • Identified only 6 people had been locked out over the past week
  • Calculated 96% success rate for the feature
  • Combined user story with data to get rounded picture of the problem
  • Discussed improvements including clearer warnings, better messaging, and alternative approaches

Result:

  • Data showed feature works smoothly most of the time
  • When it goes wrong, consequences for users are significant
  • Informed discussion about improvements with both user story and data context
  • Demonstrated approach of combining feedback and metrics for decision making

Example 3: Iterative Feature Development

Situation:
Features need to be developed iteratively, with each version tested and improved based on user feedback.

Task:
Ensure all features go through multiple iterations based on research findings.

Action:

  • Average of 2.4 iterations per feature across 32 features
  • 66% of features tested 2 or more times
  • 47% of features tested 3 or more times
  • Each iteration addresses specific findings from previous research
  • Changes validated in subsequent research rounds

Result:

  • Features continuously refined based on user feedback
  • Issues identified early and addressed before becoming major problems
  • Service quality improves with each iteration
  • Users see their feedback leading to real improvements

Example 4: Page Conditions – Continuous Iteration Over Multiple Research Rounds

Situation:
Page conditions allow users to create conditional logic for form pages, showing different questions based on previous answers. This is a complex feature that requires careful design to be both powerful and easy to use. The feature has been iterated across multiple research rounds, with each round revealing new insights and opportunities for improvement.

Task:
Continuously improve page conditions based on user feedback, addressing terminology, discoverability, usability, and technical challenges while ensuring the feature remains accessible and works for users without JavaScript.

Action:

  • Round 2 (May/June 2024): Initial testing revealed terminology issues - "Conditions" label unclear, "Add or change conditions" page not clear. Content revised for clarity.
  • Round 11 (March 2025): Found terminology issues with 'operator' confusing users, discoverability problems with conditions hidden until after saving questions, and not obvious that conditions only work if questions are on separate pages. Improved UI flow, terminology, and discoverability.
  • Round 12 (March 2025): Moved 'Filter pages by condition' to left-hand pane (previously hidden in dropdown), revealed operator and renamed to 'Condition type' (plain English). Participants navigated adding conditions successfully through exploration.
  • Round 13 (April 2025): Overall setup described as "very intuitive" and "super duper clear". Positive feedback on modularity and combining conditions. Enhanced success messages with clear CTAs, improved messaging about which pages conditions apply to.
  • Latest iteration (Current): Enhanced condition creation tools, added condition reuse features, improved previews with natural language, created overview page for managing all conditions, added warnings when deleting conditions used in multiple places, type-specific flows for different question types.
  • Addressed challenges with non-JavaScript users by ensuring progressive enhancement and fallback functionality
  • Continuously learning from each research round and applying insights to improve the feature

Result:

  • Feature evolved from basic condition creation to comprehensive condition management system
  • Most users find it easy to use - Round 13 feedback: "very intuitive", "super duper clear", "pretty straightforward"
  • Terminology improved from technical terms ('operator') to plain English ('Condition type')
  • Discoverability significantly improved - filter moved to prominent location, conditions more visible
  • Deletion warnings prevent users from accidentally breaking forms when conditions are used in multiple places
  • Type-specific flows match question behavior for better accuracy and control
  • Natural language previews reduce misunderstanding of how conditions will work
  • Feature continues to evolve - "it's never done" - with latest iteration building on positive feedback while addressing remaining user needs
  • Always learning more - each research round reveals new insights that inform the next iteration
  • Challenges with non-JavaScript users considered and addressed in design decisions

Key learning: Complex features like page conditions require multiple iterations to get right. What starts as a basic implementation needs continuous refinement based on real user feedback. The feature demonstrates that iteration is never truly complete - there are always opportunities to improve based on new insights, changing user needs, and technical considerations.
