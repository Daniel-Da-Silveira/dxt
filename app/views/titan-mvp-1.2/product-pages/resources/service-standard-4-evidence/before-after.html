Before and After: Design Changes

This page showcases key design improvements made to the Defra Forms Designer, demonstrating how user research and iterative design led to a more intuitive and accessible service. Each example includes visual comparisons and specific user feedback that drove the changes.

1. Form Overview: Organisation Details Display

Context: When users first create a new form, they add details about lead organisation and team details. Round 6 (June 2024) testing revealed usability issues with how this information was presented.

Organisation details presentation

Before: Details component

Hidden in expandable details component

  • Information hidden by default
  • Users struggled to understand what to expect when expanded
  • Rarely matched user expectations once expanded
  • Unclear how to edit organisation details

"Users struggled to understand what information to expect when it was expanded"

After: Summary list component

Visible in main body using summary list

  • Information visible immediately
  • Uses GOV.UK summary list component (purposefully designed for this)
  • Clear how to edit (edit links visible)
  • Matches user mental model

"This tested positively and users correctly identified how they could update this information"

Before: Details component expanded

Details component expanded to reveal organisation details

After: Summary list format

Organisation details in summary list format

Impact

  • Users correctly identified how to update information
  • Reduced cognitive load - information visible immediately
  • Better alignment with GOV.UK Design System patterns
  • Tested positively in user research

2. Form Overview: Form Versions Display

Context: Forms can exist in different states (draft, live). The original design presented all forms in a single table, which limited the ability to include additional information.

Form versions presentation

Before: Single table

  • All forms in one table
  • Limited ability to include additional information
  • Couldn't easily add preview links
  • Less flexible for expert users who prefer more information

After: Separate summary lists

  • Separate summary lists for draft and live forms
  • More space for additional information
  • Can include preview links and other metadata
  • Better suited for expert users

Before: Single table

Draft and live forms in a single table

After: Separate summary lists

Separate summary lists for draft and live forms

Impact

  • More flexible presentation for expert users
  • Can include preview links and additional metadata
  • Clearer separation between draft and live forms

Related design history

  • Form overview: How we present information about forms

3. Form Editor Interface Redesign

Context: Initial evaluation (March 2024) revealed significant usability issues with the component-driven interface. Users struggled to understand how to create forms.

Interface structure transformation

Before (March 2024)

Component-driven approach

  • Complex, overwhelming interface
  • Non-standardised components
  • Users struggled to understand workflow
  • Required extensive training

"Users struggled to understand how to create forms"

After (May 2024 onwards)

Task-driven approach

  • Clear hierarchical structure
  • Form overview → Page listing → Question editing
  • Intuitive navigation flow
  • Validated through 13 rounds of research

"Complete redesign validated through user testing"

Impact

  • Reduced cognitive load for users
  • Clearer task completion paths
  • Validated through 13 rounds of research (May 2024 - April 2025)

4. Question Creation Flow: Question-First Approach

Context: Round 11 (March 2025) found users were confused by the question creation flow. The interface required users to add the first list option before editing the question text due to a technical constraint. Round 12 (March 2025) tested the improved question-first approach.

5. List Options Entry: Bulk Entry and Single-Page Editing

Context: Round 8 (November 2024) found users were frustrated entering checkbox and radio options one at a time. Users requested the ability to enter options comma-separated. Round 11 found users wanted to see all options together while drafting, and disliked being redirected away from the list when adding new options.

Bulk option entry and single-page editing

Before (Round 11)

Separate pages for each option

  • Add options one at a time on separate pages
  • Redirected away from list when adding new option
  • Cannot see all options together during drafting
  • Risk of duplicating options

"You never get to see your list in the drafting stage altogether"

After (Round 12)

All options on same page

  • Question text and all options on one page
  • Bulk entry: comma-separated or line breaks
  • See complete list while drafting
  • Edit all options together

"Users appreciated entering all options on one page"

Before and After: Actual Screens

Before (Round 11): Add option on separate page

Users redirected to new page, can't see existing options

After (Round 12): All options on same page

Question text and all options visible together, bulk entry available

User feedback

{{ govukInsetText({ text: "Why can't I enter them comma-separated?" }) }}

— Participant, Round 8 (November 2024)

{{ govukInsetText({ text: "Because you have to add them one at a time and come back to this point, you never get to like see your list in the drafting stage altogether, so I could have quite easily probably duplicated them if I wasn't kind of paying attention." }) }}

— Participant 1, Round 11 (March 2025) - addressed by keeping users on the same page

Impact

  • Faster form creation - no longer need to add options one by one
  • Improved drafting visibility - users can see complete list while editing
  • Round 12 testing: "Users appreciated entering all options on one page"
  • Reduced risk of duplicate options - users can see all options together
  • More efficient workflow - question and options on same page

6. Preview Panel: Real-Time Visual Feedback

Context: The preview panel was introduced as part of the Form Editor redesign (June 2024). Previously, users had no way to see how their form would look while editing. Round 6 confirmed its value, and Private Beta Round 3 (October 2025) found it was consistently popular and essential for users.

Preview panel introduction

Before: No preview panel (Pre-June 2024)

  • Editor interface only - no visual preview
  • Users had to publish to see how form looked
  • Difficult to validate layout and content
  • No way to check formatting while editing
  • Couldn't see how form would appear to public

After: Interactive preview panel (June 2024 onwards)

  • Real-time preview panel on right side
  • Updates automatically as users edit
  • Shows exactly how form appears to public
  • Validates layout, spacing, and formatting
  • Supports all question types including repeaters

"Some users felt this feature – which is unique to Defra Forms - was essential"

8. Accessibility: Screen Reader Support

Context: Initial evaluation found accessibility issues. Needed to ensure list reordering worked with screen readers.

Screen reader announcements

Before

  • No screen reader announcements
  • Users couldn't understand current position
  • No feedback when reordering
  • Focus lost after actions

After

  • ARIA live regions for dynamic updates
  • Clear position announcements
  • Reordering feedback with new positions
  • Focus retention after actions

"Option A is currently option 2 of 4 options"

Impact

  • Screen reader users can understand current and new positions
  • All announcements include: what happened, which item moved, new position
  • Focus management allows users to continue working
  • 100% keyboard accessibility achieved

9. Form Overview: Making Form Actions Intuitive

Context: Users struggled with the original flow for creating drafts and managing form actions. The process required multiple steps through different pages, which didn't match users' mental models.

Form actions workflow

Before: Complex multi-step flow

4-step process:

  1. Open form from 'Form home' in edit mode
  2. Find and click 'Form settings' link
  3. Enter information to make form live
  4. Go back to Form home and select 'Create draft'
  • Didn't match users' mental model
  • Term 'Form home' unclear
  • Too many page movements
  • Actions hidden in settings

After: Actions on overview page

Single page with all actions:

  • Renamed 'Form home' to 'Overview'
  • Actions grouped in column layout (like Whitehall Publisher)
  • All key actions visible: create draft, make live, close live, edit draft, delete draft
  • Familiar pattern for expert users

"This feels like home" - BAU content designer

  • Reduced page movements - all actions in one place
  • Familiar pattern for users who work in Whitehall Publisher
  • Clearer page purpose - 'Overview' more intuitive than 'Form home'
  • Better alignment with user mental models

10. Page Conditions: Terminology and Discoverability

Context: Page conditions allow users to create conditional logic for form pages. Round 11 (March 2025) found terminology issues with 'operator' and discoverability problems. Round 12 and 13 refined the approach.

Page conditions evolution

Before (Round 11)

  • Technical term 'operator' confused users
  • 'Filter pages by condition' hidden in dropdown
  • Conditions appeared only after saving question
  • Not obvious conditions only work if questions on separate pages
  • Generic flow for all question types

"The word 'operator' confused people"

After (Round 13 onwards)

  • Renamed 'operator' to 'Condition type' (plain English)
  • 'Filter pages by condition' moved to left-hand pane
  • Type-specific flows: radios pre-load options, checkboxes allow multiple
  • Rich preview sentences in natural language
  • Overview page showing all conditions

"Overall setup described as 'very intuitive' and 'super duper clear'"

11. Form Builder Approach: Component-Driven to Intent-Driven

Context: The original forms designer used a component-driven approach. Initial assessment (March 2024) revealed this was confusing. The redesign moved to an intent-driven approach.

Form builder approach transformation

Before: Component-driven (x-gov)

  • Build pages by dropping individual components
  • No clear hierarchy - every modal feels equal
  • Complex component selection from three categories
  • Add components one at a time to page
  • Link-based approach for conditions

After: Intent-driven (Titan)

  • Intent-first: declare what you're adding (Question page, Guidance page)
  • Four nested levels with clear hierarchy
  • Tool narrows choices and walks you through
  • Dedicated 'Add new question' loop inside a page
  • Contextual choices revealed progressively

Key improvements

  • From component-led to intent-led: Users begin by declaring what kind of page they need, rather than hunting through a dense catalogue of components
  • Linear, guided editing: Designers move through steps with clear progression, replacing the modal-heavy approach
  • Multi-question pages: Support for adding multiple questions within a single page, with intuitive reordering
  • Consolidated page management: All page-level actions consolidated into a single Page overview screen
  • Contextual choices: Options are revealed progressively based on user intent, reducing cognitive load

Impact

  • Reduced cognitive load - clear hierarchy and progression
  • Faster form creation - guided flow instead of component hunting
  • Better user experience - matches mental model of task completion
  • Validated through 13 rounds of research
