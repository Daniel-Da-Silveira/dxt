5. How the service meets accessibility requirements

Example 1: Screen Reader Support for List Reordering

Situation:
Users need to reorder list items (radio options, checkboxes) but this functionality must work with screen readers for users who are blind or have visual impairments.

Task:
Implement accessible list reordering that works with screen readers and provides clear announcements.

Action:

  • Implemented ARIA live regions for dynamic content updates
  • Added screen reader announcements when focusing on "Move" buttons: "Button, Move [Option text]: [Up or Down], [Option text] is currently option [Position] of [Total options] options"
  • Added announcements when reordering: "The list has been reordered, [Option text] is now option [new position] of [Total options] options"
  • Implemented focus retention after reordering so users can continue making adjustments
  • Made button labels update dynamically to reflect new positions

Result:

  • Screen reader users can understand current position and new position after reordering
  • All announcements include: what happened, which item was moved, and the new position
  • Focus management allows users to continue working without losing their place
  • Documented in design history with specific code examples

Example 2: Keyboard Navigation and Skip Links

Situation:
Users who navigate using keyboards need efficient ways to move through the interface and skip repetitive content. Long forms and complex interfaces can be time-consuming to navigate with Tab key alone.

Task:
Implement comprehensive keyboard navigation with skip links to allow users to jump directly to main content areas and key features.

Action:

  • Implemented skip links throughout the interface to allow users to jump directly to specific sections
  • Added "Close preview and skip" functionality that closes preview panels and moves focus to target sections
  • Ensured all interactive elements are keyboard accessible with clear focus states
  • Implemented focus management when closing previews or modals to return focus to appropriate elements
  • Added keyboard shortcuts for common actions where appropriate
  • Tested keyboard navigation flow to ensure logical tab order

Result:

  • Keyboard users can efficiently navigate the entire interface without a mouse
  • Skip links allow users to bypass repetitive navigation and jump to main content
  • Focus management ensures users never lose their place when interacting with previews or modals
  • All features are accessible via keyboard navigation
  • Clear visual focus indicators help users understand their current position

Example 3: Accessibility Evaluation and Independent Review

Situation:
Initial evaluation (March 2024) found accessibility issues in the existing service. Needed to ensure redesign met accessibility standards.

Task:
Ensure service meets WCAG 2.2 compliance and inclusive design patterns.

Action:

  • Conducted initial heuristic and accessibility evaluation (March 2024)
  • Engaged Defra accessibility team for independent review
  • Focused on WCAG 2.2 compliance throughout redesign
  • Established accessibility by design: embedded progressive enhancement across all interaction flows to ensure the tool works without JavaScript and with assistive technology
  • Designed for parity of access: introduced a novel skip-link model enabling efficient keyboard and screen-reader navigation between editor and preview
  • Implemented keyboard navigation for all features with 100% keyboard accessibility
  • Added ARIA labels and roles for all interactive elements
  • Ensured color contrast meets standards
  • Implemented clear focus states for keyboard navigation
  • Adopted skip-link pattern across the product
  • Built accessibility testing into QA and delivery pipelines
  • Validated equivalent user outcomes across editor and preview through testing

Result:

  • Independent accessibility review completed
  • 11 features (33%) have explicit accessibility considerations documented
  • All features follow GOV.UK accessibility patterns
  • Keyboard navigation works for all features
  • Established accessibility by design: embedded progressive enhancement across all interaction flows to ensure the tool works without JavaScript and with assistive technology
  • Designed for parity of access: introduced a novel skip-link model enabling efficient keyboard and screen-reader navigation between editor and preview, ensuring equal ability to build forms
  • Achieved full parity of outcome: 100% keyboard accessibility and equivalent user outcomes established across the editor and preview, validated by testing
  • Baked-in accessibility: skip-link pattern adopted across the product; accessibility testing built into QA and delivery pipelines
